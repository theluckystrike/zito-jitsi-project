2. Script 01 — Fix UDP 10000

Why This Exists

Derek confirmed that UDP 10000 packets arrive at jitsi-00 but receive no response, even though a process holds the port. The firewall is open at every layer. The root cause is the JVB NAT harvester configuration.

The current sip-communicator.properties uses a STUN mapping harvester pointed at the local TURN server. Inside an Incus LXC container, the STUN query discovers the container's internal IP instead of the public IP 67.58.160.118. JVB advertises incorrect ICE candidates to clients. Even though packets arrive at port 10000, JVB cannot associate them with any valid ICE session and silently drops them.

How It Works

Phase 1 — Diagnostics (read-only, captures evidence): ss -ulnp ──> bindv6only ──> ip addr/route ──> sip-communicator.properties ──> JVB log harvester entries All artifacts saved to /var/log/zito-sprint2/<timestamp>/ Phase 2 — Fix (mutates config, restarts JVB): backup config ──> auto-detect internal IP ──> patch sip-communicator.properties ──> fix bindv6only ──> restart JVB Phase 3 — Verification: confirm UDP 10000 bound ──> confirm NAT harvester in JVB log shows 67.58.160.118 ──> PASS/FAIL

Core Logic — Auto-detect Internal IP

The script discovers the container's internal IP by inspecting the default route interface. No hardcoded addresses, no manual input.

$ scripts/01-fix-udp10000.sh phase2_detect_internal_ip() 
phase2_detect_internal_ip() {
    log_info "Auto-detecting container internal IP..."
    local iface ip_addr

    # Get the interface for the default route
    iface="$(ip route show default 2>/dev/null \
             | head -1 \
             | awk '{for(i=1;i<=NF;i++) if($i=="dev") print $(i+1)}')"
    if [[ -z "${iface}" ]]; then
        die "Could not determine default route interface"
    fi

    # Extract the first IPv4 address on that interface
    ip_addr="$(ip -4 addr show dev "${iface}" 2>/dev/null \
               | awk '/inet / {print $2}' | head -1 | cut -d/ -f1)"

    # Basic IPv4 validation
    if ! [[ "${ip_addr}" =~ ^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+$ ]]; then
        die "Detected IP '${ip_addr}' is not a valid IPv4 address"
    fi

    log_info "Detected internal IP: ${ip_addr}"
    echo "${ip_addr}"
}

Core Logic — Idempotent Property Setter

Properties are set idempotently — if the key exists (even commented out), the line is replaced. Otherwise, the key is appended. Safe to run multiple times.

$ scripts/01-fix-udp10000.sh phase2_set_property() 
phase2_set_property() {
    local key="$1" value="$2" escaped_key
    escaped_key="$(printf '%s' "${key}" | sed 's/[.[\*^$/]/\\&/g')"

    if grep -qE "^#?\\s*${escaped_key}" "${SIP_PROPS}" 2>/dev/null; then
        sed -i.sedtmp "s|^#*\s*${escaped_key}=.*|${key}=${value}|" "${SIP_PROPS}"
        rm -f "${SIP_PROPS}.sedtmp"
        log_info "Updated: ${key}=${value}"
    else
        printf '%s=%s\n' "${key}" "${value}" >> "${SIP_PROPS}"
        log_info "Appended: ${key}=${value}"
    fi
}

Script Logic — Idempotent Config Insertion

The script checks each file before modifying. If guest domain is already configured, it exits with code 2 (already applied). Partial states are detected and filled.

$ scripts/02-guest-domain.sh apply_meet_anonymousdomain() 
apply_meet_anonymousdomain() {
    log_info "Configuring Jitsi Meet config.js anonymousdomain..."

    if grep -q "anonymousdomain" "${MEET_CFG}" 2>/dev/null; then
        log_info "Jitsi Meet: anonymousdomain already present -- skipping"
        return 0
    fi

    # Verify the anchor line exists
    if ! grep -q "domain: '${DOMAIN}'" "${MEET_CFG}" 2>/dev/null; then
        die "Cannot find domain anchor in ${MEET_CFG}"
    fi

    # Insert anonymousdomain after the domain line
    sed -i.sedtmp \
        "/domain: '${DOMAIN}'/a\\
        anonymousdomain: '${GUEST_DOMAIN}'," \
        "${MEET_CFG}"
    rm -f "${MEET_CFG}.sedtmp"

    # Verify it was written
    if grep -q "anonymousdomain" "${MEET_CFG}"; then
        log_pass "anonymousdomain added to ${MEET_CFG}"
    else
        die "Failed to add anonymousdomain"
    fi
}
No DNS record needed. guest.jitsi-00.zitovoice.com is a Prosody-internal virtual host. It never receives direct connections. All traffic flows through the main domain. The frontend config tells the Jitsi client which XMPP domain to use for unauthenticated users, and Prosody routes internally.

4. Script 03 — Credential Rotation

Why This Exists

Three production secrets are exposed in plaintext in the public GitHub repo theluckystrike/zito-jitsi-project under raw-configs/. The repo is public. These secrets must be rotated regardless of whether the repo is made private — they are already compromised.

SecretExposed InRisk
Jicofo XMPP passwordjicofo.confImpersonate conference focus component
JVB XMPP passwordjvb-sip-communicator.propertiesImpersonate video bridge, intercept media
TURN shared secretprosody-virtualhost.cfg.luaGenerate valid TURN credentials, relay arbitrary traffic

How It Works

Phase 1 — Pre-flight: verify root ──> verify files ──> verify tools (openssl, prosodyctl) ──> backup all 4 configs Phase 2 — Generate: openssl rand -hex 16 ──> Jicofo password openssl rand -hex 32 ──> JVB password openssl rand -hex 32 ──> TURN secret All saved to /root/.jitsi-credentials-YYYYMMDD (chmod 600) Phase 3 — Prosody user registrations: prosodyctl register focus ──> prosodyctl register jvb Phase 4 — Update configs: jicofo.conf ──> sip-communicator.properties ──> prosody cfg.lua ──> turnserver.conf Each update verified with grep after write Phase 5 — Restart and verify: prosody (5s) ──> jicofo (5s) ──> jvb (3s) ──> coturn (2s) Check logs for "not-authorized" ──> must be zero
Security: The script NEVER prints secret values to stdout. New credentials are only written to the secure reference file at /root/.jitsi-credentials-YYYYMMDD (chmod 600, root-only) and to the service config files themselves. All old config files are backed up to /root/.jitsi-config-backups/<timestamp>/ with mode 700.

5. Script 04 — systemd Startup Ordering

Why This Exists

Without explicit dependencies, systemd starts Prosody, Jicofo, and JVB in parallel. Jicofo attempts to configure the JvbBrewery MUC before Prosody has initialized the internal MUC component, receives a "forbidden" error, and does not retry. The system is dead after every reboot.

What It Creates

+ /etc/systemd/system/jicofo.service.d/override.conf new file 
# Startup ordering override — Sprint 2 (theluckystrike)
# Ensures Jicofo starts after Prosody to prevent race condition
[Unit]
After=prosody.service
Requires=prosody.service

[Service]
# Brief delay to ensure Prosody XMPP listeners are ready
ExecStartPre=/bin/sleep 3
+ /etc/systemd/system/jitsi-videobridge2.service.d/override.conf new file 
# Startup ordering override — Sprint 2 (theluckystrike)
# Ensures JVB starts after Jicofo and Prosody
[Unit]
After=jicofo.service prosody.service
Requires=prosody.service

[Service]
ExecStartPre=/bin/sleep 3

Resulting Boot Sequence

Before (race condition): prosody ──parallel──> jicofo tries to configure MUC before Prosody is ready jicofo ──parallel──> "forbidden" error, no retry, system dead jvb ──parallel──> XMPP connection refused After (ordered): prosody ──[3s]──> jicofo ──[3s]──> jvb Each service starts only after its dependency is confirmed running

6. Jigasi SIP Authentication (JWT AGI)

Why This Exists

Derek asked: "Can Jitsi natively authenticate inbound SIP callers?" The answer is no — Jigasi reads the Jitsi-Conference-Room header and bridges audio without challenging the caller. But Jigasi does support a Jitsi-Auth-Token SIP header that passes a JWT straight through to Prosody for validation.

This AGI script is the Option C (JWT via SIP header) approach — the most pragmatic path that avoids building a full IVR while providing strong authentication. It reuses the existing JWT infrastructure from Sprint 1.

How It Works

Call flow: Phone caller ──PSTN──> Asterisk ──AGI──> generate-jwt-agi.py │ │ Generates JWT from CallerID + room │ Sets JWT_TOKEN channel variable ▼ SIPAddHeader(Authorization: Bearer $JWT) SIPAddHeader(Jitsi-Conference-Room: $ROOM) │ ▼ Jigasi ──passes JWT──> Prosody ──validates──> Caller joins room

JWT Generation — Zero Dependencies

The AGI script implements HS256 JWT signing using only Python's standard library. No pip install, no PyJWT, no external dependencies. HS256 is just HMAC-SHA256 over a base64url-encoded header and payload.

# jigasi/generate-jwt-agi.py generate_jwt() 
def generate_jwt(app_id, app_secret, domain, room, caller_id):
    """Build and sign an HS256 JWT for Jigasi authentication."""
    now = int(time.time())

    header = {"alg": "HS256", "typ": "JWT"}
    payload = {
        "iss": app_id,
        "sub": domain,
        "aud": "jitsi",
        "room": room,
        "iat": now,
        "exp": now + 3600,
        "context": {
            "user": {
                "name": "Phone: {}".format(caller_id),
                "affiliation": "member",
            }
        },
    }

    segments = [
        base64url_encode(json.dumps(header, separators=(",", ":"))),
        base64url_encode(json.dumps(payload, separators=(",", ":"))),
    ]

    signing_input = ".".join(segments).encode("ascii")
    signature = hmac.new(
        app_secret.encode("utf-8"), signing_input, hashlib.sha256
    ).digest()
    segments.append(base64url_encode(signature))

    return ".".join(segments)

Asterisk Dialplan — Fixed Room Pattern

; jigasi/asterisk-dialplan-example.conf Pattern 1: Fixed Room 
; Caller dials 5001 and lands in "standup-daily" room
[jitsi-fixed-room]
exten => 5001,1,Answer()
    same => n,Set(JITSI_ROOM=standup-daily)
    same => n,AGI(generate-jwt-agi.py,${CALLERID(num)},${JITSI_ROOM})
    same => n,GotoIf($["${JWT_STATUS}" != "OK"]?jwt_fail,1)
    same => n,SIPAddHeader(Jitsi-Conference-Room: ${JITSI_ROOM})
    same => n,SIPAddHeader(Authorization: Bearer ${JWT_TOKEN})
    same => n,Dial(SIP/jigasi-server,,tT)
    same => n,Hangup()

; JWT generation failed
exten => jwt_fail,1,Playback(an-error-has-occurred)
    same => n,Hangup()
