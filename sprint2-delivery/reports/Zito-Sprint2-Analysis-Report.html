3. UDP 10000 Root Cause Diagnosis

Derek's investigation eliminates the firewall at every layer. The remaining causes are all application-level — JVB is bound to the port but not processing packets correctly.

Ranked Causes

#CauseProbabilityRationale
1 STUN harvester returning wrong IP HIGH STUN server is local infrastructure — may discover container/bridge IP instead of 67.58.160.118. JVB advertises wrong ICE candidates and silently drops unmatched packets.
2 Incus proxy device without nat=true HIGH Without NAT mode, source addresses are rewritten to host bridge IP. JVB can't correlate packets with ICE sessions.
3 JVB bound to IPv6 only MEDIUM JVB may bind to udp6 only. If bindv6only=1 in the container, IPv4 packets are ignored.
4 JVB bound to wrong interface MEDIUM Multiple interfaces in the container — JVB may harvest addresses from non-routable interfaces.
5 Non-functional JVB process holding port LOWER JVB socket is bound but ICE stack not properly initialized. Possible after config change without clean restart.

Current Configuration (The Problem)

The current sip-communicator.properties uses STUN mapping harvester:

# Current — unreliable in containerized deployments
org.ice4j.ice.harvest.STUN_MAPPING_HARVESTER_ADDRESSES=turn-jitsi-00.zitovoice.com:443

When the STUN server is reachable via an internal path (same infrastructure), the harvester discovers the wrong IP — typically the container's internal IP or the host's bridge IP. JVB then advertises incorrect ICE candidates. Even though packets arrive at port 10000, JVB cannot associate them with any valid ICE session and silently drops them.

The Fix

Replace STUN harvester with explicit static NAT harvester configuration:

# Disable STUN mapping (unreliable in LXC/Incus containers)
# org.ice4j.ice.harvest.STUN_MAPPING_HARVESTER_ADDRESSES=turn-jitsi-00.zitovoice.com:443

# Explicit NAT harvester — guaranteed correct
org.ice4j.ice.harvest.NAT_HARVESTER_LOCAL_ADDRESS=<container-internal-IP>
org.ice4j.ice.harvest.NAT_HARVESTER_PUBLIC_ADDRESS=67.58.160.118

# Disable AWS harvester (not on AWS)
org.ice4j.ice.harvest.DISABLE_AWS_HARVESTER=true

Diagnostic Commands (Run Before Fixing)

# Inside the container — identify what's listening
$ ss -ulnp | grep 10000
$ cat /proc/sys/net/ipv6/bindv6only

# Check what IP JVB actually discovered
$ grep -iE "harvester|candidate|mapping|public.address" /var/log/jitsi/jvb.log | tail -80

# Check container network interfaces
$ ip addr show
$ ip route show

# On the Incus host — check proxy device config
$ incus config device show jitsi-00
$ incus config show jitsi-00 | grep -A 15 proxy

Incus Proxy Device Issue

If UDP 10000 is forwarded via an Incus proxy device without nat=true, source addresses are rewritten to the host bridge IP before delivery to the container. JVB receives the packets but the source address no longer matches the remote client. ICE correlation fails.

Without nat=true: Client (1.2.3.4:54321) ──UDP──> Host:10000 ──proxy──> Container:10000 src rewritten to 10.0.0.1 JVB sees src=10.0.0.1, not 1.2.3.4 ICE candidate mismatch → packet dropped With nat=true: Client (1.2.3.4:54321) ──UDP──> Host:10000 ──NAT──> Container:10000 src preserved as 1.2.3.4:54321 JVB correlates with ICE session → responds
Action plan: Run diagnostics first. If STUN harvester is returning wrong IP, apply the NAT harvester fix and restart JVB. If the Incus proxy device lacks nat=true, add it. Only reboot if neither fix resolves the issue — and let Derek's team handle the reboot since they own the infrastructure.

4. Guest Domain Implementation

The invite link currently fails because Prosody rejects any connection without a JWT token. The solution is a guest VirtualHost with anonymous authentication on a dedicated subdomain. Moderators continue using token-authenticated URLs. Guests join through the plain invite link on the guest domain without needing a token.

Security Model

Moderator Flow (unchanged): Token URL ──> jitsi-00.zitovoice.com ──JWT──> Prosody (token auth) ──> Room created Guest Flow (new): Invite Link ──> jitsi-00.zitovoice.com ──anon──> Prosody (guest domain) ──> Joins existing room Security gate: Jicofo ──enforces──> Room only exists while token-authenticated moderator is present ──blocks──> Guests cannot create rooms or join empty rooms

Implementation — 3 Files

File 1: Prosody — /etc/prosody/conf.avail/jitsi-00.zitovoice.com.cfg.lua

-- Add new VirtualHost for guest access
VirtualHost "guest.jitsi-00.zitovoice.com"
    authentication = "anonymous"
    c2s_require_encryption = false
    modules_enabled = {
        "bosh";
        "ping";
    }

File 2: Jitsi Meet frontend — /etc/jitsi/meet/jitsi-00.zitovoice.com-config.js

hosts: {
    domain: 'jitsi-00.zitovoice.com',
    anonymousdomain: 'guest.jitsi-00.zitovoice.com',
    muc: 'conference.jitsi-00.zitovoice.com'
}

File 3: Jicofo — /etc/jitsi/jicofo/jicofo.conf

jicofo {
    authentication: {
        enabled: true
        type: XMPP
        login-url: "jitsi-00.zitovoice.com"
    }
}
No open-access risk. The room only exists while a token-authenticated moderator is present. If the moderator leaves and no other authenticated user remains, the room closes. Guests cannot create rooms independently.

DNS Requirement

No DNS record is needed for guest.jitsi-00.zitovoice.com. This is a Prosody-internal virtual host — it never receives direct connections. All traffic flows through the main jitsi-00.zitovoice.com domain. The frontend config tells the Jitsi client which XMPP domain to use for unauthenticated users, and Prosody routes internally.

5. Jigasi SIP Dial-In — Authentication Research

Derek's question: "Is it possible to have Jitsi handle authentication of the SIP calls natively?"

Short answer: No. Jigasi does not have a built-in IVR, caller authentication system, or DTMF PIN prompt. It reads the Jitsi-Conference-Room SIP header and bridges audio to that room. No questions asked.

What Jigasi Does Support

Jigasi reads these SIP headers from inbound INVITEs (all configurable):

SIP HeaderPurpose
Jitsi-Conference-RoomTarget conference room name
Jitsi-Conference-Room-PassRoom password (bypasses lobby)
Jitsi-Auth-TokenJWT authentication token
Jitsi-User-ID-TokenAuthenticated user identity
Jitsi-Conference-Domain-BaseMulti-tenant domain base
Jitsi-VisitorVisitor mode flag

Jigasi also natively supports Jitsi's Lobby (waiting room) for SIP callers — when lobby is active, SIP callers hear "waiting for moderator review" and a web moderator can approve or reject them.

Available Options — Ranked by Effort

OptionEffortDescriptionProsCons
A. Lobby Mode Zero code Enable lobby on rooms. SIP callers enter waiting room. Moderator approves/rejects. Built into Jigasi natively. No custom code. Requires human moderator for each caller. Not scalable for unattended rooms.
B. Room Password via SIP Header Low Set room password. Asterisk sends Jitsi-Conference-Room-Pass header. Simple Asterisk dialplan change. Password hardcoded unless collected from caller via AGI first.
C. JWT via SIP Header Medium Asterisk AGI generates JWT based on CallerID lookup, sends as Jitsi-Auth-Token. Strong auth. Integrates with Jitsi's secure domain model. Minimal IVR. Requires JWT generation logic in Asterisk AGI. Caller identity verification still in Asterisk.
D. Full IVR + Conference Mapper High Deploy Conference Mapper API. Build Asterisk IVR with DTMF PIN prompt. Full commercial-grade dial-in experience. Exactly the custom system Derek wants to avoid.

Recommended Approach

Recommended: Option C (JWT via SIP Header) — The most pragmatic path if Derek wants automated authentication without building a full IVR. Asterisk does a CallerID-based lookup, an AGI script generates a JWT, and passes it as Jitsi-Auth-Token in the SIP INVITE. The "IVR" is just a Python/shell AGI script — not a full DTMF menu system. Jigasi passes the token through to Prosody, which validates it against the same JWT infrastructure already in place. 
; Asterisk dialplan — JWT passthrough
exten => _8145551212,1,AGI(generate-jwt.py,${CALLERID(num)})
exten => _8145551212,n,SIPAddHeader(Jitsi-Conference-Room:${ROOM}@conference.jitsi-00.zitovoice.com)
exten => _8145551212,n,SIPAddHeader(Jitsi-Auth-Token:${JWT_TOKEN})
exten => _8145551212,n,Dial(SIP/jigasi)

This approach reuses the existing JWT infrastructure from Sprint 1 — same app_id, same app_secret, same Prosody token validation. No new authentication system required.
