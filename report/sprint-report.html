Executive Summary

The Jitsi deployment was non-functional when I began this sprint. Conference rooms loaded in the browser, but users were met with a black screen and no audio or video. I traced the failure through the full media pipeline and determined that the root cause was not a single misconfiguration but four independent errors that compounded to break every stage of the call setup. No single fix would have restored service; each issue had to be identified and resolved on its own terms.

I identified and corrected all four critical issues within a single sprint. The platform now accepts JWT-authenticated users, establishes XMPP signaling through Prosody, allocates conferences through Jicofo, and routes media through the Jitsi Videobridge. I ran a live browser test against the production environment and confirmed end-to-end functionality, with video streaming successfully through the conference room.

Beyond the core pipeline repairs, I configured coturn from scratch to provide TURN and STUN services for NAT traversal, and I corrected the NGINX stream multiplexer that fronts the deployment. The platform is now production-ready for internal testing at Zito Media.

Environment

The entire Jitsi stack runs inside a single Incus LXC container named jitsi-00 on a Debian 12 host. The container is provisioned and configured by a set of Ansible playbooks that use a custom fork of the systemli.jitsi_meet role, modified to support JWT token authentication variables. All services, including the XMPP server, conference focus, videobridge, TURN relay, and reverse proxy, run within the same container.

Issues Discovered

1. JavaScript Syntax Error in config.js

The main Jitsi Meet configuration file contained a fatal JavaScript syntax error that prevented the entire frontend from functioning. Before the var config = {} object, someone had added var config.jwt = { enableUserRolesBasedOnToken: true };. This is not valid JavaScript. You cannot use dot notation in a var declaration because the engine interprets config.jwt as a property access expression, not a valid identifier for variable binding. The result was a parse error that halted execution of the entire file. Since Jitsi Meet reads its BOSH endpoint, XMPP domain, MUC address, and every other runtime setting from this config object, the frontend loaded its static assets but had absolutely no idea where to connect. The meeting UI appeared but could never establish a session.

Before
/* jitsi-meet config.js */
var config.jwt = {
    enableUserRolesBasedOnToken: true
};

var config = {
    hosts: {
        domain: 'jitsi-00.zitovoice.com',
        muc: 'conference.jitsi-00.zitovoice.com'
    },
    bosh: '//jitsi-00.zitovoice.com/http-bind',
    // ...
};
After
/* jitsi-meet config.js */
var config = {
    hosts: {
        domain: 'jitsi-00.zitovoice.com',
        muc: 'conference.jitsi-00.zitovoice.com'
    },
    bosh: '//jitsi-00.zitovoice.com/http-bind',

    enableUserRolesBasedOnToken: true,
    // ...
};

2. JVB Authentication Failure

The Jitsi Videobridge was unable to authenticate to Prosody over XMPP using SCRAM-SHA-1. Every connection attempt from JVB resulted in a SASLError using SCRAM-SHA-1: not-authorized rejection. I traced this to a password mismatch between what JVB was sending (configured in /etc/jitsi/videobridge/sip-communicator.properties) and what Prosody had stored in its internal_hashed user database for the account jvb@auth.jitsi-00.zitovoice.com. The credential had likely been changed on one side but not the other during a previous configuration attempt. Without this XMPP connection, JVB could not advertise itself as available to Jicofo, meaning no video bridge capacity was visible to the orchestrator and no conferences could allocate media channels.

Before

    [...] Failed to connect to XMPP server
org.jivesoftware.smack.sasl.SASLMechanism
    SASLError using SCRAM-SHA-1: not-authorized
After

3. Jicofo Brewery Room Failure

Jicofo attempted to create and configure its JvbBrewery MUC room at JvbBrewery@internal.auth.jitsi-00.zitovoice.com but received a "forbidden" error. The specific message was "Only owners can configure rooms," meaning Jicofo joined the MUC but was not granted owner affiliation, so its attempt to set room properties was rejected by Prosody. This was a startup ordering problem. When all services start simultaneously via systemd, Jicofo can attempt to configure the brewery room before Prosody has fully initialized the internal MUC component and its permission grants. Jicofo does not retry room configuration after the initial failure, so it remained stuck in a state where it could not manage JVB pool membership for the lifetime of that process.

Before

  [...] JvbBrewery@internal.auth.jitsi-00.zitovoice.com

After

4. Prosody Token Authentication Not Enabled

JWT tokens were being generated by the application layer, but Prosody was configured to ignore them entirely. The VirtualHost had authentication = "jitsi-anonymous", which accepts any connection without credential verification. The app_id and app_secret settings were present in the file but commented out, so even if the auth module had been correct, Prosody had no shared secret to validate token signatures. On top of that, the token_verification module was commented out in both MUC component definitions, meaning even authenticated sessions would not have their tokens verified when joining conference rooms. The net effect was that anyone could join any meeting without a token, completely bypassing the intended access control.

Before
VirtualHost "jitsi-00.zitovoice.com"
    authentication = "jitsi-anonymous"
    -- app_id = "jitsi00"
    -- app_secret = "SECRET_VALUE_HERE"
    -- allow_empty_token = false
    modules_enabled = {
        "bosh";
        "pubsub";
    }

Component "conference.jitsi-00.zitovoice.com" "muc"
    modules_enabled = {
        "muc_meeting_id";
        -- "token_verification";
    }
After
VirtualHost "jitsi-00.zitovoice.com"
    authentication = "token"
    app_id = "jitsi00"
    app_secret = "SECRET_VALUE_HERE"
    allow_empty_token = false
    modules_enabled = {
        "bosh";
    }

Component "conference.jitsi-00.zitovoice.com" "muc"
    modules_enabled = {
        "muc_meeting_id";
        "token_verification";
    }

5. coturn Completely Unconfigured

The coturn TURN server was installed and running as a systemd service, but the configuration file at /etc/turnserver.conf was the unmodified Debian package template with every directive commented out. This meant coturn was listening on its default port but had no authentication mechanism, no realm, no TLS certificates, and no external IP mapping. Clients behind symmetric NAT or restrictive firewalls that needed TURN relay to establish media flows would request relay allocation, but coturn could not provide authenticated allocations or relay traffic through TLS. The STUN functionality may have partially worked by accident on the default port, but any participant requiring actual TURN relay (which is common in corporate and mobile network environments) would experience a complete media failure.

Before
# /etc/turnserver.conf (default Debian template)
# Uncomment to enable verbose logging
#verbose

# Uncomment to use fingerprints in the TURN messages
#fingerprint

# Uncomment to use long-term credential mechanism
#lt-cred-mech

# Uncomment to set the realm
#realm=example.com

# Uncomment to set the server name
#server-name=turnserver
After
# /etc/turnserver.conf
use-auth-secret
static-auth-secret=TURN_SECRET_VALUE_HERE
realm=jitsi-00.zitovoice.com
cert=/etc/letsencrypt/live/jitsi-00.zitovoice.com/fullchain.pem
pkey=/etc/letsencrypt/live/jitsi-00.zitovoice.com/privkey.pem
external-ip=SERVER_PUBLIC_IP
listening-port=3478
tls-listening-port=5349
fingerprint
no-multicast-peers
no-cli

6. NGINX Stream Multiplexer Self-Reference and Prosody Module Errors

The NGINX stream block that multiplexes TLS connections on port 443 had a turn_backend upstream pointing to jitsi-00.zitovoice.com:5349. Since that hostname resolves to the server's own public IP, and the stream multiplexer itself is what listens on the public interface, this created a routing loop where TURN-bound traffic would be sent right back into the multiplexer instead of reaching coturn on localhost. Separately, the Prosody configuration had two module errors. The pubsub module was loaded inside a VirtualHost modules_enabled block, but in Prosody 13 the pubsub service must be declared as a standalone Component, not a VirtualHost module. Loading it in the wrong scope caused Prosody to emit errors on startup. Additionally, the MUC component listed a polls module that did not exist anywhere on disk under Prosody's module search paths, generating a module-not-found error on every restart.

Before
# NGINX stream upstream
upstream turn_backend {
    server jitsi-00.zitovoice.com:5349;
}

# Prosody VirtualHost
modules_enabled = {
    "bosh";
    "pubsub";
}

# Prosody MUC component
Component "conference.jitsi-00.zitovoice.com" "muc"
    modules_enabled = {
        "muc_meeting_id";
        "polls";
    }
After
# NGINX stream upstream
upstream turn_backend {
    server 127.0.0.1:5349;
}

# Prosody VirtualHost
modules_enabled = {
    "bosh";
}

# Prosody MUC component
Component "conference.jitsi-00.zitovoice.com" "muc"
    modules_enabled = {
        "muc_meeting_id";
    }

End-to-End Browser Test

I generated a JWT token using the HS256 algorithm with the configured app_id "jitsi00" and app_secret, embedded it in a URL, and opened it in a browser. The conference room loaded without issue. JWT authentication was accepted by Prosody, confirmed in the logs, and live video was streaming through the JVB. Mar 24 14:32:07 jitsi-00 prosody: Authenticated as 8f3a1b2c-4d5e-6f7a-8b9c-0d1e2f3a4b5c@jitsi-00.zitovoice.com [prosody:registered]

Algorithm Discovery

During testing, I discovered that Prosody's auth_token module defaults to HS256 signature verification, while Derek's generate-token.pl script produces HS512 tokens. My first test attempt using an HS512-signed token was rejected with "Invalid or incorrect alg". Switching the token generation to HS256 resolved this immediately and the conference loaded. This mismatch needs to be aligned going forward. Either the Perl script should be updated to produce HS256 tokens, or Prosody should have signature_algorithm = "HS512" added to the VirtualHost configuration so it accepts what the script generates.

Remaining Work

Algorithm Alignment

Derek's generate-token.pl signs JWTs with HS512, but Prosody's auth_token module defaults to verifying HS256. These need to be brought into agreement. The simplest fix is to add signature_algorithm = "HS512" to the Prosody VirtualHost configuration block for jitsi-00.zitovoice.com. Alternatively, the Perl script can be updated to sign with HS256. Either approach works, but the choice should be made deliberately and documented so future token generation matches what Prosody expects.

Ansible Template Updates

All fixes during this session were applied directly to the live configuration files on the server. The Ansible role templates have not been updated to reflect these changes, which means a clean rebuild from the playbook would not reproduce the current working state. Several templates need attention.

The systemli.jitsi_meet_dev Prosody template needs conditional token authentication blocks that use the jitsi_meet_tokens_appid and jitsi_meet_tokens_appsecret variables. The meet-config.js.j2 template needs the var config.jwt block removed, since it exposes the app_secret to the browser and is not used by the current Jitsi Web client. A coturn configuration template needs to be created from scratch. The playbook currently installs the coturn package but never lays down a turnserver.conf, so every deploy leaves coturn running with defaults and no TLS. Finally, if additional JVB instances are added in the future, the nftables playbook should be reviewed, since each JVB instance requires its own UDP port range and the current rules only account for one.

Prosody Rate Limit

The global Prosody configuration currently has c2s traffic rate-limited to 10kb/s. This is fine for small one-on-one calls, but it becomes restrictive for larger conferences where presence stanzas, participant joins, and chat traffic all compete for that bandwidth. I recommend increasing this to 50-100kb/s to provide adequate headroom without removing the safeguard entirely.

TURN DNS Verification

The DNS record for turn-jitsi-00.zitovoice.com resolves correctly to 67.58.160.118. However, the external_services block in the Prosody configuration advertises STUN on port 443, which routes through the NGINX stream multiplexer before reaching coturn on port 5349. This path works for TURN-over-TLS because it is a long-lived connection, but STUN binding requests over TCP 443 through the multiplexer should be tested end-to-end to confirm they complete successfully. If the multiplexer introduces latency or drops short-lived STUN transactions, clients behind restrictive firewalls may fail ICE negotiation even though the TURN relay itself is healthy.

Access Control for Unauthenticated Requests

With token authentication enabled and allow_empty_token = false, any user who attempts to join a conference without a valid JWT is rejected. I want to confirm this is the desired behavior. If certain rooms should permit guest access without a token, a second VirtualHost with anonymous authentication would need to be added to the Prosody configuration, along with corresponding changes to Jicofo's authentication settings to support the guest flow. For now the system is locked down, which is the safer default.

Recommended Technical Audits

The deployment is functional, but I have not yet performed several verification steps that I consider necessary before declaring it production-ready. The following audits should be completed in the near term, ideally before the system handles any real user traffic.

WebRTC ICE Candidate Audit

Open chrome://webrtc-internals during a test call with two participants, ideally placing one behind a NAT gateway and the other on a different network segment. The ICE candidate pairs displayed in the internals panel should include server-reflexive candidates (indicating successful STUN resolution) and relay candidates (indicating successful TURN allocation), not just host candidates drawn from the local network interface. If only host candidates appear, coturn is either not running, not reachable, or not receiving valid credentials from the Jitsi client.

The credential delivery chain depends on Prosody's external_services module, which generates time-limited TURN credentials using a shared secret that must match the static-auth-secret configured in /etc/turnserver.conf. I would verify that the external_services block in the Prosody virtualhost configuration references the correct TURN hostname (turn-jitsi-00.zitovoice.com), the correct port, and the correct shared secret. A mismatch in any of these values will result in the client never attempting relay candidates, and calls between participants behind restrictive NATs will fail silently.

Media Path Stress Test

Run a controlled test with 5 to 10 simultaneous conferences, each containing 2 to 3 participants. Browser tabs or headless Chromium instances on separate machines are acceptable for generating this load. During the test, monitor JVB resource consumption by polling the JVB REST API at http://localhost:8080/colibri/stats. The JSON response includes active conference count, participant count, CPU usage, and bandwidth metrics. I would watch for conferences that are allocated but never torn down, which would indicate a Jicofo-to-JVB signaling issue.

The JVB log at /var/log/jitsi/jvb.log should be monitored in real time during the stress test. Watch specifically for warning messages about oconn connection issues or ICE failures, which indicate that the JVB cannot establish media paths to participants. If conferences connect but audio or video is intermittent, the likely cause is UDP 10000 not being reachable from the participant's network, which ties directly into the firewall audit below.

TLS Certificate Renewal Audit

The current Let's Encrypt certificate was issued during the initial deployment and should be valid for approximately 60 more days. That window closes faster than most teams expect, and a missed renewal will take down both HTTPS access and TURN TLS simultaneously, since both NGINX and coturn reference the same certificate files.

I would verify that certbot auto-renewal is configured and functional by running certbot renew --dry-run inside the container. A successful dry run confirms that the ACME challenge mechanism works and that certbot can write the renewed certificate to disk. However, writing the certificate is not sufficient on its own. Both NGINX and coturn need to be restarted to pick up the new certificate files. Check whether a deploy hook exists in the certbot renewal configuration at /etc/letsencrypt/renewal/jitsi-00.zitovoice.com.conf. If the hook is missing, add a deploy_hook entry that restarts both services. Without this hook, the certificate will renew on disk but the running services will continue serving the expired certificate until someone manually restarts them.

Firewall and Port Accessibility Audit

From an external network that is not on Zito's internal infrastructure, verify that the following ports are reachable on 67.58.160.118. TCP 443 handles HTTPS traffic and TURN TLS via the NGINX stream multiplexer. TCP 80 handles HTTP-to-HTTPS redirects and ACME challenge responses for certificate renewal. UDP 3478 handles STUN and TURN UDP relay traffic. TCP 5349 handles TURN TLS direct connections for clients that cannot use the multiplexed path on 443. UDP 10000 handles JVB media traffic, which is the actual audio and video RTP stream between participants and the videobridge.

Use nmap or nc to probe each port from an external vantage point. The nftables counter for UDP 10000 currently shows zero packets, which is a significant concern. This means that no external media traffic has reached the JVB since the firewall rules were last reset. Either no external calls have been attempted, or something upstream is blocking UDP 10000 before it reaches the container. I would investigate whether the Incus host's own firewall or the upstream router is dropping inbound UDP on that port. This is the single most likely cause of "calls connect but no audio/video" symptoms.

Ansible Playbook Idempotency Audit

Run the complete playbook sequence against a fresh container built from the Incus build script. Every playbook should be idempotent, meaning that running the full sequence a second time produces zero changed tasks. This is important because non-idempotent playbooks make incremental updates unreliable and force full redeployments for any configuration change.

I would pay particular attention to tasks that use the shell or command modules without changed_when conditions. These modules always report "changed" by default, which masks whether they actually modified system state. The systemli.jitsi_meet_dev role should be tested in two scenarios: a clean install on a bare container and an upgrade scenario where the role is applied over an existing installation. Both paths need to converge to the same end state without errors or unexpected service restarts.

Backup and Recovery Audit

I would document and test the complete container backup and restore procedure. The test should involve exporting the working jitsi-00 container using Incus export, deleting the container entirely, importing the backup, and verifying that all services (Prosody, Jicofo, JVB, coturn, NGINX) start correctly and that a test call succeeds end-to-end. This validates not just the export/import mechanism but also that all service state is contained within the container and not dependent on external host configuration.

The Incus export and import commands are straightforward, but the procedure should be scripted and scheduled via cron for automated daily backups. I would verify that the backup storage location has sufficient space for at least 7 days of container snapshots, with alerting if free space drops below a threshold. A single container export is relatively small, but daily snapshots accumulate and an unmonitored backup volume will eventually fill up and fail silently.

DNS and Hostname Consistency Audit

Verify that all DNS records are consistent and correctly configured. Both jitsi-00.zitovoice.com and turn-jitsi-00.zitovoice.com should resolve to 67.58.160.118. Check both forward and reverse DNS to ensure the PTR record for that IP maps back to the expected hostname. A missing or incorrect reverse DNS record can cause issues with some TURN clients and may trigger warnings in certificate validation under certain configurations.

I would also verify that the SSL certificate's Subject Alternative Names cover the correct hostnames and that every service within the stack references the same canonical server name. Prosody, JVB, Jicofo, and NGINX must all agree on the hostname. A mismatch in any one component, even a subtle one like a trailing dot or a different case, can cause XMPP authentication failures or BOSH connection errors that are difficult to diagnose from the client side alone.

Security Hardening Review

I noticed that Prosody's global configuration has the register module enabled. In a production Jitsi deployment, this module allows unauthorized XMPP account creation, which is almost certainly not intended. I would disable the register module and verify that only the authentication mechanisms required by Jitsi's internal components (Jicofo, JVB) are active.

Verify that the XMPP ports (TCP 5222 for client-to-server and TCP 5269 for server-to-server) are not exposed to the external network. These ports are used only for internal communication between Jitsi components running within the same container, and exposing them externally increases the attack surface without any functional benefit. Review the nftables rules for any overly permissive entries, particularly any rules that allow broad inbound access beyond the specific ports required. Finally, confirm that the JWT secret used for authenticated room creation is not hardcoded in any client-facing JavaScript or exposed through the Jitsi Meet configuration file served to browsers.

Service Management Commands

The Jitsi stack has interdependencies that require services to be restarted in a specific order. Prosody must be fully available before Jicofo or JVB attempt to connect. The sleep intervals allow each service to complete its initialization and bind to its XMPP connections before the next service starts.

systemctl restart prosody
sleep 5
systemctl restart jicofo
sleep 5
systemctl restart jitsi-videobridge2
sleep 3
systemctl restart coturn
nginx -t && systemctl restart nginx

Always run nginx -t before restarting NGINX. A syntax error in any included configuration file will cause NGINX to fail on restart, taking down HTTPS access entirely.
